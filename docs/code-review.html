Executive Summary

Verdict: Clean, well-structured early-stage sequencer with strong foundations and several concrete areas for improvement.

The codebase follows the project's own CLAUDE.md conventions consistently: thin global hooks, context-object state management, modular decomposition. The architecture is sound for a norns script of this complexity. However, there are meaningful gaps in musicality features, some performance concerns in the step engine, and several modularity opportunities that would pay dividends as the project grows.

The review below is organized by concern: architecture, modularity, readability, performance, reliability, musicality, playability, test coverage, and prioritized recommendations.

1 — Architecture Assessment
Strengths
  • Context object pattern is consistently applied. All state flows through ctx, making the code testable and free of hidden globals.
  • Thin entrypoints — re_kriate.lua (55 lines) and re_kriate_seamstress.lua (63 lines) are exactly what CLAUDE.md prescribes: only global hooks, delegation to modules.
  • Voice abstraction is clean: nb_voice, midi, and recorder all conform to the same play_note/note_on/note_off/all_notes_off interface without a formal class system.
  • Dual-platform support — norns and seamstress share the core (app, sequencer, track, grid_ui) and only diverge at entrypoint/voice/screen layers.
  • Remote API is well-designed: transport-agnostic dispatch, validation helpers, OSC transport as a separate layer. The list_paths() introspection is a nice touch.
Concerns
  • app.lua couples UI + init + params: It creates params, connects grid, sets up metro, AND handles screen redraw. This is the single file most likely to become unmaintainable.
  • No formal voice interface contract: The duck-typed interface works now but has no documentation or validation. A voice missing play_note would fail silently at runtime deep in sequencer.lua.
  • grid_ui.lua mixes rendering + input + nav logic in one 233-line file. The nav_key function requires a delayed require of sequencer to toggle play/stop, which is a code smell.
  • Sequencer creates unbounded coroutines: ratchet spawns a new clock.run for each ratcheted step with no cancellation tracking. Fast sequences can accumulate zombie coroutines.

Key Modularity Recommendations
  1. Split app.lua into: app/init.lua (params, grid, metro setup), app/screen.lua (norns screen redraw), app/input.lua (key/enc handlers). The init function is doing too much — it should orchestrate, not implement.
  2. Split grid_ui.lua into: grid/render.lua, grid/input.lua, grid/nav.lua. The delayed require of sequencer in nav_key would resolve naturally if nav delegates to a callback on ctx.
  3. Remove duplicated extended page tables in grid_ui.lua — EXTENDED_PAGES/EXTENDED_REVERSE are defined both as M.EXTENDED_* and as local EXTENDED_*. Use only the module exports.
  4. Define a voice interface spec as a comment or small validation function. Current duck-typing relies on convention alone.

4 — Performance Analysis
Timing Context

At 120 BPM with 1/16th division, step_track fires every 125ms. At 180 BPM, every ~83ms. The step function must complete well under this to avoid clock drift. The norns Lua runtime is single-threaded — any blocking in a clock coroutine delays all other coroutines.

IssueSeverityDetailsRecommendation
Ratchet coroutine leak High sequencer.lua:101-108: Each ratchet spawns a clock.run but the coroutine ID is never stored or cancelled. If the sequencer is stopped mid-ratchet, these coroutines linger until they naturally complete. At high ratchet counts (7) with fast divisions, coroutines accumulate. Store ratchet coroutine IDs on the track/ctx. Cancel them in sequencer.stop().
Per-step table creation in step_track Medium sequencer.lua:64-67: Creates a new vals = {} table every step, populating it via loop. On norns (LuaJIT), this is fast but generates GC pressure at high step rates. Pre-allocate a vals table per track on ctx, reuse it each step. Eliminates allocation in the hot path.
Grid metro fires at 30fps unconditionally Low app.lua:66-73: The grid metro runs at 30fps even when nothing changes. The grid_dirty flag prevents unnecessary redraws, but the metro itself still ticks. This is standard norns practice and acceptable. The dirty check makes it O(1) when idle.
Seamstress screen metro at 15fps Low re_kriate_seamstress.lua:47: Redraws every frame regardless of changes. No dirty check. Add a screen_dirty flag similar to grid_dirty.
recorder.clear uses O(n²) algorithm Low recorder.lua:60-68: table.remove in a forward-scan loop shifts all subsequent elements on each removal. O(n²) worst case. Test-only code, so not critical. Could be fixed with a reverse scan or filter-copy approach.
Linear param name validation in remote API Low api.lua:29-33: check_param does a linear scan of PARAM_NAMES (8 entries) for every API call that involves a param. Convert to a hash set. Marginal gain but trivial to implement.
Load Time Assessment

Startup: The script loads approximately 14 Lua modules via require. Each is small (<400 lines). On norns hardware (ARM Cortex-A72), this completes in well under 100ms. musicutil.generate_scale is the heaviest operation at init and it runs once.

Grid connection: grid.connect() is async on norns. The metro starts immediately. Grid display appears within one metro tick (33ms) after connection.

Overall: Init performance is excellent for a norns script. No concerns about load time.

6 — Musicality Review

Core question: Does this sound and feel like kria?

Kria's musical identity comes from: independent per-parameter loop lengths creating polymetric patterns, intuitive scale quantization, smooth clock division, and the interplay between tracks. Let's evaluate each aspect.

FeatureStatusAssessment
Per-param loop lengths Done Each of the 8 params (trigger, note, octave, duration, velocity, ratchet, alt_note, glide) has independent loop_start/loop_end. This is the heart of kria and it works correctly. Polymetric sequences emerge naturally.
Direction modes Done 5 modes (forward, reverse, pendulum, drunk, random). Applied per-track but not per-param. In original kria, each param lane can have its own direction. This limits polymetric possibilities.
Scale quantization Partial 14 scales available. Root note is configurable. However: the to_midi function hardcodes 7 degrees per octave. Pentatonic Major/Minor (5 degrees), Blues (6), Whole Tone (6), and Chromatic (12) will produce incorrect note mapping. The degree selector (1-7 on grid) doesn't match these scales.
Clock division Done 7 divisions from 1/16 to whole note, per-track. Uses clock.sync for tight timing. Division is a param, not per-step — matches original kria.
Ratchet Done Per-step ratchet count (1-7) subdivides the note duration evenly. Musically effective for adding rhythmic intensity. Coroutine management is the only concern.
Glide/Portamento Partial Grid UI for setting glide values exists. sequencer.lua calls voice:set_portamento(). But neither nb_voice nor midi voice implements set_portamento. It's a no-op in practice. The MIDI voice would need CC 5 (portamento time) + CC 65 (portamento on/off).
Alt-note Done Adds a secondary note degree offset. Wraps around the scale correctly via modular arithmetic. Good for creating melodic variation without changing the base pattern.
Velocity expression Basic 7 velocity levels (0.15-1.0). Mapped linearly. The jump from 0.15 to 0.30 at the bottom is proportionally large, making subtle dynamics at low velocity hard to achieve. An exponential curve would feel more natural.
Pattern storage Missing No pattern save/recall. Original kria has multiple pattern slots. This is a significant musical limitation — you lose everything on script reload.
Pattern morphing / probability Missing No trigger probability, no pattern interpolation. These are common in modern sequencers and would add musical depth.
Swing / groove Missing No swing parameter. All divisions are quantized to straight time. Even a simple swing amount (delay even-numbered steps) would significantly improve groove feel.
Transposition Missing No per-track or global transposition. The root note param changes the scale globally, but there's no way to transpose a pattern up/down by scale degrees during performance.
Scale Quantization Bug Detail

The SCALE_NAMES in app.lua include "Pentatonic Major", "Pentatonic Minor", "Blues Scale", "Whole Tone", and "Chromatic". But scale.lua:to_midi does:

local idx = (3 + oct_offset) * 7 + degree

That 7 is hardcoded. For pentatonic scales (5 degrees per octave), this formula skips notes and wraps incorrectly. For chromatic (12 degrees per octave), the grid's 7-row display can only access 7 of 12 pitches anyway.

Fix: Either (a) restrict scale choices to 7-degree scales only, or (b) compute degrees per octave dynamically and adjust the grid display row count. Option (a) is simpler and arguably more kria-like.

7 — Playability Review
Grid Interaction Quality

The grid UI is functional but has several playability gaps compared to original kria:

InteractionQualityNotes
Step toggle (trigger page) Good Toggle on press, 4-track view, playhead at brightness 15. Standard kria behavior.
Value editing (note/oct/dur/vel) Good Bar graph display with 7 rows. Press sets value directly. In-loop steps have distinct brightness. Playhead highlights well.
Loop editing Decent Hold loop button + press start + press end. Works but no visual feedback showing the current loop region while editing. User can't see what they're changing until they release the loop button.
Extended page toggle Good Double-tap page button to access ratchet/alt_note/glide. Nav highlights the primary button when on extended page. Intuitive.
Track select Good Grid buttons 1-4 on nav row. Encoder E1 on norns. Immediate feedback.
Mute per track Missing UI Mute is implemented in sequencer (and available via remote API) but there's no grid button or norns key to toggle mute. Users must use the remote API or modify code.
Direction per track Missing UI Direction is implemented in sequencer but no grid or norns UI to change it. Only accessible via remote API.
Loop region visibility Partial On trigger page: loop region shown at brightness 2. On value pages: steps outside loop are dimmer. But: when loop is shorter than 16, the out-of-loop area is just "dark" with no indication of where the loop boundaries are.
Screen UI (norns) Minimal Shows title, track/page, play state. No step position visualization, no parameter values, no track overview. The screen is dramatically underutilized — most of the 128x64 display is empty.
Keyboard (seamstress) Basic Space/r/1-4/qwety for essential controls. No keys for extended pages, mute, direction, or loop editing.
